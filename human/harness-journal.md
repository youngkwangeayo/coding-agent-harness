# 하네스 저널

`coding-agent-harness`를 사람과 AI가 함께 만들어온 기록. 무엇을 왜 그렇게 정했는지, 그리고 아직 안 닫힌 질문을 사람 눈높이로 남긴다.

---

## 1. 어떻게 만들었나

- **방식**: AI가 한 블록씩 초안 제안 → 사람이 승인/수정 → 확정 → 다음 블록. 처음부터 전체를 쓰지 않고 점진적으로 쌓음.
- **토대 합의**: AI가 계획을 주도하고, 사람은 승인·검증한다.

## 2. CLAUDE.md 결정 요약

### 협업 원칙
- 별도 섹션 2개로 띄우는 A안 vs 한 섹션 + 소제목 B안을 비교 → **B안 채택** (같은 주제는 한 우산 아래, 확장에 강함).
- 소제목을 형식어(`행동 규약`)가 아니라 **원칙 문장 그대로** 올림 — "주도"가 제목에서 살게.
  - `AI가 계획을 주도하고, 사람은 승인·검증한다`
  - `요구자도 사람이다 — 요구 자체도 틀릴 수 있다` (사람의 요구도 틀릴 수 있다는 전제)

### 작업 수칙
- **완료의 정의** = 검증까지. + 검증 가능한 목표로 변환(버그 재현 테스트 먼저 → 통과).
- **변경은 수술하듯** — 요청에 직결된 것만 건드림 + 내 변경이 만든 죽은 것만 정리(둘은 한 원칙의 양면이라 병합).
- **운영·인프라는 재현 가능하게** — 개발은 코드가 남지만 운영은 IaC처럼 명령을 남겨야 함.
- **최소한으로** — 요청 안 한 기능·추상화·유연성 금지(과설계 방지).

### 문서화
- 도메인 문서(내부에 쌓는 살아있는 지식, 추후 agentic 수집용 단일 소스).
- 외부 노출용 도메인 문서(다른 서비스가 전체 코드 안 훑고 연동할 계약 표면).
- ADR — 택한 이유보다 **폐기한 길과 그 이유**가 더 중요(막다른 길 재방문 방지).

### 기록·트래킹
- 세 채널, 역할 안 겹침: **git**(무엇/왜 바꿈) · **`docs/PROGRESS.md`**(현재 상태 스냅샷) · **`docs/PM-TASKS.md`**(보고 업로드용).
- 작업 규모에 맞춤 — 사소한 변경은 git만.
- **연속성 루프**: PROGRESS.md를 멈추기 직전 쓰고, 세션 시작 시 먼저 읽음.

## 3. 중간에 바로잡은 것

- **과잉 게이트 제거**: "단계마다 승인"이 "AI 주도" 간판과 모순 → 게이트는 *계획 승인 / 결과 검증* 두 지점으로.
- **삭제 모순 해소**: "승인 없이 삭제 금지" ↔ "죽은 것 정리"가 충돌 → 비가역·외부·남의 파일 삭제만 승인, 내 산출물 정리는 자율.
- **Karpathy CLAUDE.md 흡수**: Simplicity / 수술하듯(minimal-diff) / 검증 가능한 목표만 흡수. 이미 있던 것(불확실하면 묻기, dead-code 규칙)과 "caution 우선" 톤은 배제(우리 AI 주도와 충돌).

## 4. 열린 질문 — 가 vs 나 (영속 스킬/에이전트 생성)

**런타임 서브에이전트 띄우기**는 이미 자율(도구 사용). 문제는 **영속 스킬/커맨드/에이전트 파일을 알아서 만들지**다.

- **(가) 현 상태 — 제안 후 승인.** 영속 산출물 생성은 AI가 제안하고 사람이 승인. 과설계 방지에 강하지만, 반복 작업 자동화가 느림.
- **(나) 적극 자동화 — 트리거 규칙.** "같은 작업 N회 반복 시 AI가 알아서 스킬/커맨드로 만들고 통보". 빠르지만 과설계 위험을 경계로 관리해야 함.

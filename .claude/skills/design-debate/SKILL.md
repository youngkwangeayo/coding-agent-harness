# 토론 리뷰 (design-debate)

설계·구현·결정을 **토론 리뷰**한다. 목적은 양쪽 편향을 *동시에* 잡는 것 — 구현이 한쪽으로 쏠려 더 나은 길을 놓쳤거나 앞선 결정에 어긋났는지, 그리고 검증이 실제 제약을 무시한 책상물림 비판은 아닌지.

## 먼저: 할지·얼마나 할지 판단한다 (적응적)

토론의 *모양*은 상황에 맞춘다 — 이건 고정 파이프라인이 아니다.

- **사소하거나 명백한 건 전부 돌리지 않는다.** 한 줄짜리·취향 수준이면 "여긴 토론 불필요"라고 밝히고 가볍게 본다. (CLAUDE.md: 최소한으로, 과설계 금지)
- **쟁점이 크고 되돌리기 어렵거나, 설계 방향·트레이드오프가 걸린 것**일수록 풀세트를 돌린다.
- 변호인 패스 수, 원문 드릴다운 시점, 대상 범위는 **그때그때 판단**한다.

## 절대 안 바뀌는 원칙 (이게 이 도구의 핵심)

규율은 즉흥에 맡기면 드리프트한다. 아래는 토론을 돌리는 한 *반드시* 지킨다:

1. **독립성** — `design-reviewer`와 `design-advocate`를 **서로의 출력을 안 보게** 띄운다(한 메시지에서 병렬). reviewer가 변호에 설득당해 유효 지적을 접거나, advocate가 발견사항에 맞춘 변명을 짜내는 걸 차단.
2. **reviewer는 비평가가 아니다** — "잘 되었나"를 판정한다. 잘된 점을 반드시 적고, 지적을 "문제 아님"으로 닫을 수 있다.
3. **증거 우선, 편 없음** — arbiter는 리뷰어도 advocate도 추인하지 않고 증거가 어느 쪽을 지지하는지만 본다. 다툼이 크면 원문을 직접 확인한다.
4. **ADR 씨앗 산출** — 제약 때문에 "유지"로 판정된 건 *왜 그 길을 폐기했는지*를 남긴다(CLAUDE.md: ADR — 폐기한 길과 이유).

## 절차

1. **대상 확정** — 무엇을(diff/모듈/PR/결정/설계) 리뷰하는지 한 줄로 명시. 비어 있으면 현재 브랜치 변경분(`git diff master...HEAD`)이나 직전 산출물을 잡고 무엇을 잡았는지 밝힌다. 관련 ADR·design·retro 문서 위치를 파악해 둔다.
2. **독립 병렬 실행** — `design-reviewer`(잘된 점 + 발견사항 P1~P3)와 `design-advocate`(원의도·논리 + 불가피했던 제약 + 정직한 약점)를 동시에 띄운다. 각자에 대상과 관련 문서 위치를 전달한다.
3. **심판** — 두 결과를 `design-arbiter`에 넘겨 항목별 판정(고친다 / 유지+ADR씨앗 / 바꾼다)을 받는다. arbiter가 "타깃 재질의 필요"를 표시하면 그 쟁점만 `design-advocate`에 1회 되물어 보강 후 다시 넘긴다(무한 루프 금지).
4. **종합 보고** — 사람에게: ① 확정된 **잘된 점**, ② **변경 목록** P1~P3(각 항목 valid/constrained/better 판정·근거), ③ **ADR 씨앗**, ④ **다음 행동 제안**.

## 주의

- 여기까지는 **판정과 변경안**이다. 실제 코드 수정은 **사람 승인 후** 별도로 한다 (CLAUDE.md: 계획 승인 게이트).
- 세 서브에이전트는 모두 읽기 전용. 토론 중 코드를 바꾸지 않는다.
- 사실대로 보고한다 — arbiter가 "전체적으로 잘 됐다"고 하면 그렇게 보고하고, 억지로 고칠 거리를 만들지 않는다.
- 대상을 *내가 만든 것*으로 토론할 땐, 오케스트레이터인 나의 **확증편향**을 보고에 명시한다(독립 에이전트가 완화하나 제거하진 못함).
